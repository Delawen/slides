Hybrid Cloud

The topic today is "Integration Processes for Serverless Hybrid Cloud". You may have notice how software engineers love to use buzzwords to make it sound smart. Let's start by understanding what these words mean.

What do you mean by "Integration Processes"?

When you need components to interact

Create (work)flows and automations

Complex software architectures are made of several components you have to interconnect, like a database, a user interface, an analysis tool,... The pieces of software that connects them (in purple) are the middleware, the "glue", the orchestration between those components. These interactions between different unit operations are what make your usecase work.

Usually you define a flow that defines how each component interact with the rest. This flow can be linear or circular, split into several threads, have conditional steps,... and that's what you have to implement to make your usecase work. These are what are called integrations.

They are usually run more than once and that's why you have more than one process.

What do you mean by "Hybrid Cloud"?

When you have mixed components

different providers, different languages, different paradigms,...

A cloud is an internet-based computing in which most of your computing capacity is remote. That cloud allows the sharing of data-processing tasks, centralized data storage, and online access to computer services or resources. It can be very useful for "as a Service" features like Software as a Service (SaaS), Platform as a Service (PaaS), Function as a Service (FaaS),... which provides of "on-demand" functionality that enables the supporting infrastructure to be powered down and not incurring charges when not in use.

An Hybrid Cloud is a cloud that has a variety of providers, services, paradigms,... Some of those components can be free and open source as opposed to others having a restricting license. Being able to make such different components interact with each other is usually a complex task, specially when each of them "speaks" a different language, protocol and/or format.

What do you mean by "Serverless"

Serverless means literally a cloud with no server running behind. But that may be confusing.

Serverless means you use the cloud with only "as a Service" capabilities, powering up only the time needed. You will not have idle servers wasting resources (not only money but also electricity).

So the point of this talk is to show you a way of create flows between components that need to interact using a mixed paradigm of components with "as a Service" features.

If it is still not clear: to create workflows of data between components trying to use a sustainable software. Where sustainable means easy to setup and maintain and using the fewest resources as possible.

"Glue"/Middleware

Integration Frameworks, can help us orchestrate and automate the interaction between different applications, APIs, third party services or any other software piece we may have to connect.

A proper integration tool should provide us with the following features: transformation, integration patterns and connectors to existing protocols and components.

Let's start with transformations. When we connect different components of an architecture, they rarely speak the same languages or, on this case, data formats. Some components will output an xml that has to be fed to the following component on a json form. Maybe we even need to add or remove some attributes on that json data. An integration framework allows us to define what is the mapping between the output of one component and the input of the following so we can forget about the explicit implementation. The less code we have to maintain, the better.

Now, being able to transform data on each step is not enough if we can‚Äôt connect to (and from) the specific component we need. Our ideal integration framework should offer support for common protocols like ftp, http, jdbc,‚Ä¶ Also it should offer support to connect to common components like a mail server, messaging services, atom,‚Ä¶ We could claim even that no integration tool would be good if it doesn‚Äôt also support specific well known services like being able to send a message through a Telegram bot or store information on Elastic Search. Being able to seamlessly connect from one component to the next without having to worry about the specifics of their interfaces is what distinguishes an average integration tool from a good integration tool.

And finally, not all flows will be lineal. Some of the steps will require broadcasting, some of them will require conditional flowing. Some will require waiting the output of different components to conflate the data. These action patterns are something that have been studied for a long time. And as with software development patterns, you can classify them and study them to create better integrations. You can find them as "Enterprise Integration Patterns".

Apache Camel

Compared with Lego

Apache Camel is not only one of the most active projects inside the Apache Software Foundation, it is also the lightest and most complete integration framework available. And on top of it, it is also Free and Open Source Software!

Camel is already an old actor on the integration world. It has support for hundreds of components, protocols and formats. Some of these protocols come very handy allowing the user, for example, to connect to any REST API that they need.

Camel uses its own DSL, a simplified language to define easily the workflows step by step.

Camel is also available in Knative. This means, we can use it on a serverless environment, making sure the orchestration between services runs and escalates properly.

Let's see an example
