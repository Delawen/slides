But for each transformation, you may lose data (golden cat eyes). So you may have the same data but with less fields or attributes. We have to make sure we don't lose anything important on the transformation. And try to converge in having the same data in all formats, even if the format is different. Same with protocols.

Some universal rules all communities should adopt

  • Use (resolvable) URI’s to identify things
  • Records in a database (exposed via WFS)
  • Metadata in a catalogue
  • References to codelist items
  • Re-use ontologies and codelists where possible

Use Case 1: Spatial Catalogue to "Open Data" portal

The quotes are because a Spatial Catalog may also be "open data".
  • Requires a transformation from iso19139 to DCAT
  • May need alternative protocols then CSW, eg OAI, RSS Challenges
  • Paginated synchronisation, if the process takes long time, the index may be changed in between
  • If multiple portals are crawled by search engines, you may end up with multiple hits of the same dataset in the search engine, if each portal mints a new URI
This is really the first DCAT mention in the whole presentation?

Use Case 2: WFS to search engine

  • Have a webpage created for every WFS-record (no ajax)
  • Feed the webpages to the search engines with a sitemap or by linking from list pages.
  • Use schema.org microdata to have the search engine crawl the content as structured data
  • Provide value to end-users who are finding your records on the search engines (ability to provide feedback, links to other resources)
